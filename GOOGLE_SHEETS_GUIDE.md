# Google Sheets 补货自动计算工具——最终简化版说明

## 一、最终使用方式

按照mentor的要求，不再使用“补货参数”工作表，也不再自动抓取交期、物流天数和箱规。

在“每周判断快速下单”中，每一行只需要人工填写五项：

| 列 | 内容 | 填写方式 |
|---|---|---|
| A | 判断周的周一日期 | 手填，例如2026/7/20 |
| B | SKU | 手填，必须与产品工作表名称一致 |
| C | 交期天数 | 手填，可以因SKU和当周情况不同而变化 |
| D | 截单到入库上架天数 | 手填 |
| L | 箱规 | 手填 |

脚本自动填写：

| 列 | 自动结果 |
|---|---|
| E | 预计入库日期A |
| F | 断货预警天数，固定21 |
| G | 断货预警日期B |
| H | 本周是否需要下单，是或否 |
| I | 补货天数C |
| J | 平均控制日销D |
| K | 按箱规向上取整的补货数量E |
| M | 预计断货日期S |
| N | 补货覆盖结束日期F |
| O | 处理状态或错误原因 |
| P | 最后计算时间 |

脚本不会覆盖C、D和L。原来的“补货参数”工作表已经不再参与计算，可以保留、隐藏或人工删除。

## 二、计算规则

### 1. 预计入库日期A

`A = 判断周周一 + 交期天数C列 + 截单到入库上架天数D列`

### 2. 断货日期S和预警日期B

脚本读取与SKU同名产品页的全部日期、控制销量和控制库存，识别下一轮有效断货：

- 判断周库存大于0：寻找后面第一次控制库存小于或等于0的日期。
- 判断周库存小于或等于0，但未来入库让库存重新大于0：跳过当前缺货段，从恢复后寻找下一次断货。
- 判断周库存小于或等于0，而且预测范围内始终没有恢复：显示“持续缺货”，不使用普通公式编造补货量。

预警天数固定为21：

`B = 预计断货日期S - 21天`

### 3. 是否需要下单

找出预警日期B所在周的周日：

- 如果预计入库日期A晚于或等于该周周日，H列为“是”。
- 如果预计入库日期A早于该周周日，H列为“否”。

### 4. 需要下单时计算数量

`补货天数C = 预计入库日期A - 预警日期B + 14`

`覆盖结束日期F = 预计断货日期S + 补货天数C`

控制日销D取断货日期S至覆盖结束日期F之间的平均控制日销。

`原始补货量 = 补货天数C × 平均控制日销D`

`最终补货量E = 按L列箱规向上取整`

## 三、新产品和新SKU如何处理

不需要补货参数表，也不需要刷新SKU名单。

新增产品只要满足以下条件，就可以直接使用：

1. 新建一个与SKU完全同名的产品工作表。
2. 产品页包含“识别SKU”“控制销量”“控制库存”表头。
3. “识别SKU”列中存在与工作表名称一致的SKU。
4. 日期列连续，控制销量和控制库存公式已经向后延伸。

然后在快速判断表新一行填写A、B、C、D、L，脚本会在本次计算时重新读取所有工作表并自动找到新SKU。无需改代码、无需运行刷新参数。

如果只有SKU名称，但还没有建立产品页或公式，脚本会停止并在O列说明问题，不会套用其他SKU的数据。

## 四、为什么可以持续使用2—4年

脚本没有写死2026年，也没有写死产品页最后一行。每次运行都会读取产品页当前所有有日期的数据，因此以后增加2027、2028、2029、2030甚至更晚的数据时，不需要修改代码。

为了持续使用，产品页必须同步维护：

- 日期继续逐日向下延伸，不能缺少判断周周一。
- “控制销量”公式继续向下延伸。
- “控制库存”公式继续向下延伸。
- 新的预计入库正确写入“库存信息”，并反映到控制库存。
- 不要出现重复日期。

系统已经验证2027跨年、2028闰年2月29日、2029和2030日期计算。

## 五、预测范围不足时如何处理

“暂时没有找到断货”不等于“无需下单”。如果产品页只做到几个月后，数据长度不足，旧逻辑可能错误输出“否”。最终版已经修正：

- 只有预测范围足够覆盖预计入库日、21天预警期和一周缓冲时，才能确认H列为“否”。
- 如果预测范围不够，H列留空。
- O列显示产品页目前预测到哪一天，以及至少需要延伸到哪一天。

这样，即使明年或后年的数据暂时还没建立，脚本也不会假装知道结果。以后公式延伸后，重新计算即可自动得到新结果。

## 六、日期安全规则

- A列必须填写星期一日期，因为业务规则按周判断。
- 日期统一按照当前Google工作簿时区读取，避免星期一被误判成星期日。
- 支持Google日期值、`yyyy/mm/dd`、`yyyy-mm-dd`和`m/d/yyyy`。
- 闰年和跨年通过日期运算自动处理。
- 早于当前周的历史周会被阻止，防止旧周混入当前下单判断。

如果明明填写2026/7/20仍出现星期一错误，说明Apps Script中仍保留旧版代码。应删除旧版补货代码，只保留最终版。

## 七、数据安全规则

以下情况不会输出一个看起来精确但不可靠的补货量：

- A、B、C、D、L任一手填项缺失。
- 交期不是大于或等于0的整数。
- 截单到入库上架天数或箱规不是大于0的整数。
- SKU产品页不存在或表头结构错误。
- 判断周周一在SKU页中没有对应日期。
- 控制库存出现空值、错误值或公式未延伸。
- 产品页存在重复日期。
- 补货平均区间缺少控制日销。
- 控制日销出现负数。
- 预测范围不足以判断。
- 当前持续缺货且没有未来入库使库存恢复。
- 正常预警窗口已经错过。

所有问题都会写入O列，H和K不会错误地给出正常结论。

## 八、首次更新脚本

1. 先使用Google表格副本测试。
2. 点击“扩展程序 → Apps脚本”。
3. 找到当前存放补货代码的脚本文件。
4. 删除旧版补货代码，粘贴最终版 `replenishment_apps_script.gs`。
5. 不要同时保留两套补货代码。
6. 保存。
7. 在函数下拉框选择 `setupReplenishmentTool`。
8. 点击“运行”。
9. 返回Google表格刷新页面。
10. 检查表头已经变成C、D、L“手填”，F“固定21”。

## 九、日常使用

1. 新增一行。
2. A列填写本周或未来某周的星期一。
3. B列填写SKU。
4. C列填写交期。
5. D列填写截单到入库上架天数。
6. L列填写箱规。
7. 等待脚本自动计算，或点击“补货工具 → 计算当前选中行”。
8. 先检查O列，再采用H和K列结果。

## 十、已经完成的回归验证

- DEMO-FAST标准案例可以输出预计入库日、预警日和整箱补货量。
- DEMO-LONG在预测范围充足且没有断货时，可以安全输出本周无需下单。
- DEMO-RECOVERY在判断周已经缺货但未来库存恢复时，会跳过当前缺货段。
- 2026/7/20等星期一日期可以正常处理，不会因时区被误判为星期日。
- 2027/1/4、2028/2/28、2029/1/1、2030/1/7均正确识别为星期一。
- 2030年完整补货案例可以正常计算并按箱规向上取整。
- 新SKU在超过1500行的长产品表中仍可自动识别。
- 预测范围不足时不会错误输出“否”。

## 十一、仍需和mentor确认的一个边界

当前平均控制日销默认包含覆盖结束日期F，即S和F首尾两天都参与平均，因此使用的日期数量为“补货天数C + 1”。如果mentor要求只取严格的C个日期，需要把代码中的 `INCLUDE_COVER_END_DATE_IN_AVERAGE` 改为 `false`。
